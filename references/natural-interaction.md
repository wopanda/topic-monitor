# 自然语言交互原则

## 产品定位

`topic-monitor` 不是给用户背命令的工具。

它的产品体验应该是：
- 用户用自然语言说想监控什么
- agent 帮用户完成：配置、试跑、展示样稿、确认是否开启定时发布
- 用户默认不需要先看脚本路径、环境变量、命令行参数

一句话：
> **用户要的是一个“主题监控能力”，不是一份 shell 教程。**

## 交互原则

参考“少而准、先结果后设置”的设计方式：

1. **先理解用户要监控什么**
   - 不先甩配置文件路径
   - 不先要求用户自己改 JSON

2. **先帮用户跑出第一版结果**
   - 第一轮重点是让用户看到成品
   - 不先让用户学习内部结构

3. **跑完后用自然语言追问**
   - 追问应该像产品对话，不像命令行提示
   - 推荐口径：
     - `这版我先给你跑出来了。要不要顺手给你设成每天自动发？`
     - `如果你想每天收这份监控，我可以继续帮你接上定时发布。`

4. **只有当用户明确要自己接管时，才暴露命令和路径**
   - 例如用户问：`怎么自己改配置？`
   - 或者：`具体命令给我`

## 不推荐的交互

不要默认这样说：
- `第一步编辑配置`
- `第二步运行命令`
- `第三步看输出`
- `export BOCHA_API_KEY=...`

这些内容可以保留在 README 里，但**不应该成为默认前台交互**。

## 推荐 agent 行为

### 用户说：
- `帮我监控 OpenClaw 的应用案例`

### 推荐响应路径：
1. agent 直接理解主题
2. agent 帮用户准备配置 / 试跑
3. agent 先把日报给用户看
4. agent 再问：
   - `这版先给你跑出来了。要不要我顺手给你设成每天自动发？`

## CLI 资料的定位

命令、路径、配置文件、环境变量这些内容：
- 应保留在 README / references 里
- 供“想自己维护的人”查看
- 不应成为默认产品交互文案

# 定时发布推荐方案

## 默认产品策略

`topic-monitor` 默认分两层：

1. **内容生产层**
   - 负责搜索、筛选、生成日报 markdown
   - 主入口：`scripts/topic-monitor-run.sh`

2. **定时发布层**
   - 负责按时间触发、读取当日日报、调用消息发送能力
   - 推荐由外层调度器 / workflow / agent 完成

## 为什么不建议把发送硬编码进 skill 本体

原因：
- 搜索生成和消息投递是两种不同职责
- 换渠道时（Feishu / WeCom / Telegram）不应该重写 skill 核心逻辑
- 出错时更容易区分是“日报生成失败”还是“发送失败”
- 更接近现有成功案例：**外层调度 + 外层发送**

## 推荐默认流程

### 第一次使用
1. 用户先跑通一次日报生成
2. 用户先看样稿，确认：
   - 主题对不对
   - 输出风格对不对
   - 来源过滤是否过严 / 过松
3. **第一次跑通后，再问用户是否要开启定时发布**

建议问法：
- 要不要把这个主题监控设成每天自动发？
- 如果要，我就按配置里的时间和时区接上外层调度。

### 用户确认开启定时发布后
1. 外层调度器按 `schedule.time` + `schedule.timezone` 每天触发一次
2. 先执行 `scripts/topic-monitor-run.sh` 生成当日日报
3. 再由外层读取 `output/YYYY-MM-DD-主题监控日报.md`
4. 最后调用消息发送层发到目标渠道

## 默认值口径

如果用户未指定具体时间，但明确说“开启定时发布”，推荐默认值：
- 时间：`09:00`
- 时区：`Asia/Shanghai`
- 发送方式：外层调度 + 外层发送

## 不建议的方案

不建议默认采用：
- skill 内部自己等待到点
- skill 内部硬编码调用某一个渠道 SDK
- 未经用户确认就自动开始每天发送

## 适合 agent 的执行口径

当用户说：
- “先帮我试跑” → 只生成日报，不自动发
- “可以，帮我每天发” → 再接定时发布层
- “先看看样子” → 先出日报样稿，再问是否开启定时
